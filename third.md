Использование ASP.NET Core позволяет построить надёжное серверное приложение, рассчитанное на одновременную работу множества пользователей и администраторов.

Основным языком программирования серверной части является C#. Данный язык обладает строгой типизацией, развитой объектно-ориентированной моделью и поддержкой асинхронного программирования. Строгая типизация позволяет обнаруживать значительную часть ошибок ещё на этапе компиляции, что особенно важно при разработке системы с большим количеством связанных сущностей и бизнес-правил.

Для хранения данных используется Microsoft SQL Server. Выбор данной системы управления базами данных обусловлен её надёжностью, поддержкой транзакций, индексов, ограничений целостности и удобной интеграцией с платформой .NET. В приложении хранятся связанные данные: пользователи, идентификационные номера, голосования, варианты ответов, голоса, refresh-токены и записи аудита.

Для взаимодействия приложения с базой данных используется Entity Framework Core 8. Данная ORM-библиотека позволяет описывать структуру данных в виде C#-классов и конфигураций, выполнять запросы к базе данных с помощью LINQ, а также управлять изменениями схемы базы данных через механизм миграций. В проекте ограничения таблиц, связи, индексы и типы столбцов описываются через Fluent API в методе `OnModelCreating`, что позволяет централизованно управлять схемой данных. Для защиты от повторного голосования в таблице `Votes` определён составной уникальный индекс по полям `UserId` и `PollId`:

```csharp
modelBuilder.Entity<Vote>(entity =>
{
    entity.HasKey(e => e.Id);
    entity.HasOne(e => e.User)
          .WithMany(u => u.Votes)
          .HasForeignKey(e => e.UserId)
          .OnDelete(DeleteBehavior.Restrict);
    entity.HasOne(e => e.Poll)
          .WithMany(p => p.Votes)
          .HasForeignKey(e => e.PollId)
          .OnDelete(DeleteBehavior.Restrict);
    entity.HasOne(e => e.Option)
          .WithMany(o => o.Votes)
          .HasForeignKey(e => e.OptionId)
          .OnDelete(DeleteBehavior.Restrict);

    // Защита от повторного голосования
    entity.HasIndex(e => new { e.UserId, e.PollId }).IsUnique();
});
```

Архитектура приложения построена по принципам слоистой архитектуры с разделением ответственности. Presentation-слой содержит контроллеры API, обрабатывающие HTTP-запросы: `AuthController`, `PollsController`, `VotesController`, `AdminController`, `AuditController` и `UploadController`. Application-слой представлен сервисами бизнес-логики, где реализованы сценарии создания голосований, обработки голосов, управления пользователями, формирования отчётов и аудита: `AuthService`, `PollService`, `VoteService`, `JwtService`, `EmailService`, `ExcelExportService`, `AuditService`. Infrastructure-слой включает репозитории для работы с базой данных через Entity Framework Core (`AppDbContext`), а также реализации сервисов генерации JWT-токенов, отправки электронной почты и экспорта данных. Такое разделение уменьшает связанность кода и делает систему более удобной для тестирования и сопровождения.

Для защиты API используется JWT-аутентификация. После успешного входа или регистрации пользователь получает access token, который передаётся в заголовке `Authorization` при последующих запросах. Также в системе применяется refresh token, позволяющий обновлять пользовательскую сессию без повторного ввода учётных данных. Разграничение доступа реализовано на основе ролей: в системе выделяются обычные пользователи и администраторы. Первый зарегистрировавшийся пользователь автоматически получает права администратора; последующие администраторы назначаются при помощи секретного кода.

Для отправки пользователям электронных писем с кодом подтверждения email применяется библиотека MailKit. Она обеспечивает надёжную работу с протоколом SMTP и позволяет формировать сообщения как в HTML, так и в текстовом формате. Для хеширования паролей используется алгоритм BCrypt через библиотеку BCrypt.Net-Next, что исключает хранение паролей в открытом виде.

Для реализации клиентской части был выбран чистый JavaScript (vanilla JS) в сочетании с HTML5 и CSS3. Для оформления интерфейса применяется фреймворк Bootstrap 5, обеспечивающий единообразие карточек, таблиц, форм, модальных окон и элементов навигации. Клиентская часть организована в виде набора HTML-страниц: список голосований, страница голосования, панель администратора, журнал аудита, история голосований, профиль, подтверждение email, вход и регистрация.

Для обмена данными в реальном времени и мгновенного отображения результатов голосования используется SignalR. Клиентская часть устанавливает WebSocket-соединение с хабом `PollHub`, присоединяется к группе, соответствующей идентификатору голосования, и получает обновления без необходимости ручного обновления страницы.

Для документирования и проверки API используется Swagger/OpenAPI. Swagger позволяет просматривать список доступных эндпоинтов, структуру запросов и ответов, а также выполнять тестовые запросы к серверу на этапе разработки и отладки. Для формирования отчётов по результатам голосований в формате Excel применяется библиотека ClosedXML.

Таким образом, выбранный стек технологий позволяет реализовать безопасную, функциональную и структурированную систему онлайн-голосований. ASP.NET Core, MS SQL Server и Entity Framework Core обеспечивают надёжную серверную основу, JavaScript и Bootstrap позволяют создать удобный веб-интерфейс, а SignalR обеспечивает интерактивность при отображении результатов в реальном времени.


Описание алгоритма работы программного продукта

Разрабатываемое веб-приложение представляет собой сервис онлайн-голосований, состоящий из клиентской и серверной частей. Пользователь взаимодействует с системой через веб-интерфейс, реализованный на HTML, CSS и JavaScript. Клиентская часть отправляет HTTP-запросы к серверному API, получает ответы в формате JSON и обновляет состояние интерфейса. Серверная часть, реализованная на ASP.NET Core, выполняет аутентификацию, авторизацию, обработку бизнес-логики, взаимодействует с базой данных MS SQL Server через Entity Framework Core и возвращает результат выполнения операции.

Общий алгоритм обработки запроса в системе можно представить следующим образом. Пользователь выполняет действие в интерфейсе, например открывает список голосований, проголосовывает или создаёт новое голосование. Клиентская часть формирует HTTP-запрос и передаёт его на сервер. Если endpoint требует авторизации, middleware ASP.NET Core проверяет наличие access token в заголовке `Authorization`. При успешной проверке токена в контекст запроса добавляются данные текущего пользователя: идентификатор, адрес электронной почты, полное имя и признак администратора.

Далее запрос передаётся в контроллер, который обращается к сервису бизнес-логики. Сервис выполняет валидацию входных данных, проверку прав доступа, обращается к репозиториям и при необходимости изменяет состояние доменных сущностей. Репозитории работают с контекстом Entity Framework Core и сохраняют изменения в базе данных. После завершения операции сервер возвращает результат клиентской части, а интерфейс обновляет карточки, таблицы, уведомления и связанные данные.

Одним из ключевых сценариев системы является регистрация и аутентификация пользователей. При регистрации пользователь отправляет на сервер свои данные: полное имя, адрес электронной почты, пароль, идентификационный номер и опциональный код администратора. Сервер проверяет уникальность email, валидность идентификационного номера в таблице `PersonalIds`, вычисляет хеш пароля с помощью BCrypt, генерирует шестизначный код подтверждения email и сохраняет пользователя в базе данных. Код отправляется на указанный адрес через SMTP. При входе пользователя сервер проверяет email и пароль, после чего генерирует пару access token и refresh token. Access token используется для доступа к защищённым endpoint, а refresh token применяется для обновления сессии без повторного ввода пароля.

Ключевой фрагмент алгоритма получения идентификатора текущего пользователя из JWT-токена представлен в листинге 3.1.

Листинг 3.1 – Получение идентификатора пользователя из JWT-токена
```csharp
private Guid? GetUserId()
{
    var userIdClaim = User.FindFirst(ClaimTypes.NameIdentifier)?.Value
                   ?? User.FindFirst("sub")?.Value;
    if (Guid.TryParse(userIdClaim, out var userId))
        return userId;
    return null;
}
```

После входа пользователь может выполнять действия в рамках своей учётной записи. Для защиты от повторного голосования в базе данных используется составной уникальный индекс по полям `UserId` и `PollId`. Благодаря этому пользователь не может проголосовать в одном и том же голосовании дважды.

Рассмотрим алгоритм управления голосованиями. Администратор создаёт голосование через панель администратора. На сервер передаются название, описание, тип голосования (один вариант, несколько вариантов или да/нет), варианты ответов, изображения, настройки видимости результатов, признак real-time обновления, а также дата и время начала и окончания. Сервер проверяет корректность дат: дата начала и окончания не может быть в прошлом, а дата окончания должна быть позже даты начала. При создании голосование получает статус `Draft` (черновик), что позволяет внести правки до публикации. Для типа «Да / Нет» варианты создаются автоматически.

После публикации голосование переходит в статус `Active` (активное) и становится доступным для голосования. При изменении статуса или сроков действия сервер фиксирует событие в журнале аудита. Алгоритм публикации голосования и создания записи аудита представлен в листинге 3.2.

Листинг 3.2 – Публикация голосования и фиксация действия в аудите
```csharp
var success = await _pollService.PublishPollAsync(id);
if (!success) return BadRequest(new { message = "Ошибка публикации. Возможно, голосование уже опубликовано или не существует." });

await _auditService.LogAsync(userId, email ?? "", "PUBLISH", "Poll", id.ToString(), "Голосование опубликовано");
return Ok(new { message = "Голосование опубликовано" });
```

При голосовании пользователь выбирает вариант или несколько вариантов ответа. Сервер получает идентификатор пользователя из токена, проверяет, что голосование активно и текущее время находится в пределах заданного срока, убеждается, что пользователь ещё не голосовал, и проверяет соответствие типа голосования количеству выбранных вариантов. Для голосований с одним вариантом или типа «Да / Нет» допускается только один выбор, для множественного выбора — один или несколько. Все переданные идентификаторы вариантов проверяются на принадлежность к текущему голосованию.

После успешной проверки в базе данных создаются записи голосов. Если у голосования включён режим real-time, сервер через SignalR отправляет обновлённые результаты всем клиентам, подключённым к данному голосованию. Алгоритм расчёта результатов представлен в листинге 3.3.

Листинг 3.3 – Расчёт результатов голосования
```csharp
var counts = await _voteRepository.GetResultsAsync(pollId);
var total = counts.Values.Sum();

var results = poll.Options.Select(o =>
{
    var count = counts.ContainsKey(o.Id) ? counts[o.Id] : 0;
    var percentage = total > 0 ? Math.Round((double)count / total * 100, 2) : 0;
    return new VoteResultDto
    {
        OptionId = o.Id,
        OptionText = o.Text,
        Count = count,
        Percentage = percentage
    };
}).ToList();
```

В системе предусмотрены гибкие настройки видимости результатов. Результаты могут быть всегда видны, видны только после голосования или скрыты до окончания голосования. При запросе результатов сервер проверяет соответствующие правила и возвращает данные только в том случае, если пользователь имеет на это право.

Администратор может закрыть голосование досрочно, продлить срок окончания или удалить голосование. При закрытии статус меняется на `Closed`, при удалении запись удаляется из базы данных вместе с каскадно связанными сущностями. Все действия администратора фиксируются в журнале аудита.

Важной частью работы системы является проверка прав доступа. Защищённые endpoint требуют наличия токена. Для операций управления голосованиями дополнительно проверяется, что текущий пользователь является администратором. Алгоритм проверки прав администратора представлен в листинге 3.4.

Листинг 3.4 – Проверка прав администратора
```csharp
private bool IsAdmin()
{
    return User.FindFirst("is_admin")?.Value?.ToLower() == "true";
}
```

Сохранение изменений в репозиториях выполняется непосредственно через контекст Entity Framework Core. Репозитории отвечают за получение и изменение сущностей, а сервисы — за бизнес-логику и координацию операций. Такая организация позволяет централизованно управлять транзакциями и обеспечивает целостность данных.

Обработка ошибок организована на уровне контроллеров и сервисов. В сервисах используются проверки бизнес-правил и выброс исключений `ArgumentException` при нарушении условий, например при некорректных датах. Контроллеры преобразуют исключения в соответствующие HTTP-ответы: `BadRequest` при ошибках валидации, `Unauthorized` при отсутствии токена, `Forbid` при недостаточных правах и `NotFound` при отсутствии запрашиваемой записи.

Отдельные функции системы работают по общей схеме. Модуль аудита фиксирует важные действия пользователей: создание и удаление голосований, публикацию, закрытие, продление, голосование, вход в систему, загрузку файлов и назначение прав администратора. Администратор может просматривать журнал аудита в отдельной панели. Модуль экспорта позволяет скачать результаты голосования в виде Excel-файла, содержащего название, тип, статус, общее количество голосов и распределение по вариантам.

Таким образом, алгоритм работы программного продукта строится по единой схеме: пользователь выполняет действие в интерфейсе, клиентская часть отправляет запрос на сервер, сервер проверяет токен и права доступа, передаёт запрос в соответствующий сервис, выполняет бизнес-логику и операции с базой данных, после чего результат возвращается в интерфейс. Такая организация обеспечивает безопасность, единообразие и предсказуемость обработки операций в системе онлайн-голосований.


Обработка ошибок в программе

При разработке веб-приложения для голосований особое внимание уделяется обработке ошибок, так как система работает с важными данными: учётными записями пользователей, голосованиями и результатами. Ошибки могут возникать при вводе некорректных данных, отсутствии прав доступа, обращении к несуществующей записи, нарушении бизнес-правил или сбоях при работе с базой данных.

Первый уровень обработки ошибок связан с валидацией входных данных на стороне клиента. На страницах входа и регистрации JavaScript проверяет обязательные поля, корректность формата email, минимальную длину пароля и совпадение типов данных. Например, при регистрации проверяется, что пароль содержит не менее шести символов, а поле email соответствует шаблону адреса электронной почты. Если данные не проходят проверку, форма не отправляется, а пользователь получает понятное сообщение об ошибке рядом с соответствующим полем.

Второй уровень связан с контролем бизнес-правил на сервере. Например, нельзя опубликовать уже опубликованное голосование, отредактировать закрытое голосование, проголосовать в неактивном голосовании или повторно, а также создать голосование с датой окончания, предшествующей дате начала. Такие ситуации не являются техническими ошибками, но нарушают правила предметной области, поэтому для них сервисы выбрасывают исключения `ArgumentException`, которые контроллеры преобразуют в ответ со статусом 400.

Третий уровень связан с авторизацией и аутентификацией. Если пользователь не передал JWT-токен или токен недействителен, сервер возвращает статус 401, а клиентская часть перенаправляет пользователя на страницу входа. Если пользователь авторизован, но не является администратором и пытается получить доступ к панели управления, возвращается статус 403. Например, обычный пользователь не может создать голосование или просмотреть журнал аудита.

Для единообразной обработки ошибок на клиентской стороне используется централизованная функция `api`, которая оборачивает все HTTP-запросы. Она перехватывает ответы сервера и преобразует их в понятные сообщения. Благодаря этому страницы не содержат повторяющихся блоков обработки ошибок, а пользователь всегда получает ошибку в предсказуемом формате.

Листинг 3.5 – Обработка ошибок API на клиентской стороне
```javascript
if (response.status === 401) {
    localStorage.removeItem('accessToken');
    localStorage.removeItem('refreshToken');
    window.location.href = '/login.html';
    return;
}

if (!response.ok) {
    let errorData;
    const contentType = response.headers.get('content-type');
    if (contentType && contentType.includes('application/json')) {
        errorData = await response.json().catch(() => ({ message: `HTTP ${response.status}` }));
    } else {
        const text = await response.text().catch(() => '');
        errorData = { message: text || `HTTP ${response.status}` };
    }
    throw new Error(errorData.message || `HTTP ${response.status}`);
}
```

На стороне клиентского приложения ошибки API отображаются с помощью всплывающих уведомлений (toast). Ошибки валидации показываются рядом с соответствующими полями формы, отсутствие прав доступа отображается как сообщение «Нет доступа», а при истечении срока действия access token выполняется перенаправление на страницу входа. При возникновении непредвиденной ошибки сервер возвращает статус 500. Такие ошибки не раскрывают пользователю внутренние технические детали приложения, однако записываются в журнал сервера для дальнейшего анализа разработчиком.

Таким образом, обработка ошибок в системе онлайн-голосований построена на нескольких уровнях: проверка входных данных на клиенте, контроль бизнес-правил на сервере, проверка прав доступа, обработка HTTP-статусов в централизованном API-клиенте и отображение понятных сообщений пользователю. Такой подход повышает устойчивость приложения и делает взаимодействие пользователя с системой более предсказуемым.
